# Project Constitution

This document defines the permanent engineering and design rules for ProjectHub. These rules act as the project's source of truth and should not be changed without deliberate discussion.

---

# Product Philosophy

ProjectHub is a collaboration platform.

It is **not**:

* A College ERP
* A Student Management System
* An Admin Dashboard

The product is inspired by the workflow of platforms such as GitHub, LinkedIn, Notion, and Discord, while serving the specific needs of college innovation.

---

# Design Philosophy

Every screen should feel:

* Premium
* Editorial
* Minimal
* Modern
* Elegant
* Calm
* Timeless

The interface should resemble a funded startup product rather than a traditional student project.

---

# Architecture Principles

* Single Responsibility Principle
* Feature-based organization
* Reusable components
* Clear separation between frontend and backend
* API-first communication
* Scalable folder structure
* Consistent naming conventions

---

# Development Rules

* Never redesign approved pages.
* Never change the design language without discussion.
* Never change the architecture without discussion.
* Build reusable components whenever appropriate.
* Prefer maintainability over shortcuts.
* Complete one milestone before starting the next.
* Review, commit, and push after each milestone.

---

# UI Principles

* Cream-first interface
* Large typography
* Comfortable whitespace
* Rounded cards
* Thin borders
* Minimal shadows
* Smooth and purposeful animations

---

# Coding Principles

* Keep components focused on a single responsibility.
* Avoid duplicate UI components.
* Organize features logically.
* Keep business logic outside UI components.
* Write readable code before clever code.

---

# Git Workflow

Main Branches

* main
* develop

Feature Branch Format

* feature/<feature-name>

Bug Fix Branch Format

* bugfix/<issue-name>

Commit Format

* feat:
* fix:
* refactor:
* style:
* docs:
* chore:
* test:

---

# Quality Checklist

Before every commit, verify:

* Design matches the approved design system.
* Code is reusable where appropriate.
* Naming conventions are followed.
* No unnecessary duplication exists.
* Responsive behavior has been considered.
* Architecture remains consistent.

---

# Long-Term Goal

ProjectHub should be developed to a standard suitable for:

* Professional portfolio presentation
* Technical interviews
* Open-source collaboration
* Real-world scalability

Every implementation decision should support that goal.
